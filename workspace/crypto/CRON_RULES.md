# Crypto Cron Rules

更新时间：2026-03-27

## 目标
把加密巡检拆成多条 cron 时，保证它们分工清楚、口径一致、不互相打架。

---

## 一、角色定义

### 1）主巡检（主脑）
负责：
- 市场结论
- 账户结论
- 宏观 / 事件结论
- 最终操作建议
- 原因 + 证据链

不负责：
- 高频盯盘
- 短时间内重复提醒同一件事

输出要求：
- 中等长度
- 结论完整
- 必须明确“符合用户规则 / 不符合用户规则”

### 2）高频轻巡检（哨兵）
负责：
- 真实仓位异动
- 浮盈亏明显变化
- 关键位突破 / 失守
- 风险暴露突然上升

不负责：
- 长篇分析
- 宏观深度解释
- 重复输出主巡检已经讲清楚的完整逻辑

输出要求：
- 1~3 句
- 只说变化、影响、动作

### 3）宏观 / 事件巡检（雷达）
负责：
- 未来 24h / 72h 高影响事件
- ETF 流异常
- 重大监管节点
- 事件是否改变当前仓位建议

不负责：
- 盘口细节
- 高频价格波动
- 对仓位做过细技术分析

输出要求：
- 短
- 只讲事件和影响

## 二、共享上下文

三条 cron 默认共享：
- `/home/zbb/.openclaw/workspace/crypto/WORKFLOW.md`
- 用户真实 OKX 仓位
- 用户个人交易规则
- 当前关注标的：BTC / ETH / DOGE

### 当前用户规则
- 风格：短线
- 单笔最大可接受回撤：10%~15%
- 10x：首次特殊试验仓，不视为常态
- 市场乐观可容忍修复
- 市场悲观优先止损
- 默认不补仓条件：
  1. 主流偏弱
  2. DOGE 未明显转强
  3. 只是亏损扩大但证据未改善
  4. 高杠杆试验仓
  5. 高影响事件临近且方向未明

## 三、避免打架的规则

### 规则 1：主巡检优先级最高
同一主题若都能发，主巡检负责完整结论。

### 规则 2：高频轻巡检只报变化
如果主巡检刚发过完整结论，轻巡检只有在以下情况才允许再发：
- 浮盈亏明显扩大 / 收窄
- 关键位被破
- 仓位变化
- 风险等级明显升级

### 规则 3：宏观巡检只在影响建议时发
只有在以下情况才发：
- 新增高影响事件
- 事件进入 24h 窗口
- ETF 流明显异常
- 监管事件新增
- 事件会改变当前仓位建议

### 规则 4：同一主题 2 小时内不重复长提醒
除非：
- 风险升级
- 仓位变化
- 关键位失守
- 宏观层新增重大变化

---

## 四、触发条件

### A. 主巡检
满足任一即可发：
1. BTC / ETH / DOGE 趋势变化
2. 动作建议变化
3. 资金风格变化
4. 主流关键位突破 / 失守
5. 仓位变化
6. 浮盈亏明显扩大 / 收窄
7. 新的高影响事件改变建议

### B. 高频轻巡检
只盯：
1. 当前持仓浮盈亏明显扩大
2. 当前持仓浮盈亏明显收窄
3. 关键位被有效突破 / 失守
4. 风险暴露上升
5. 当前建议从“观察”变成“该处理了”

默认不发：
- 小波动
- 结论没变化
- 原区间震荡

### C. 宏观 / 事件巡检
1. 未来 24h 新增高影响事件
2. 未来 72h 出现新的重要节点
3. ETF 流异常
4. 重大监管事件出现
5. 这些事件会改变当前仓位建议

---

## 五、输出格式

### 1）主巡检格式
固定四段：
1. 市场结论
2. 账户结论
3. 宏观 / 事件结论
4. 建议 + 原因 + 证据

要求：
- 明确写“符合用户规则 / 不符合用户规则”
- 宏观层失败时必须标注“宏观/事件层证据不足”或“宏观/事件层暂不可用”

### 2）高频轻巡检格式
固定一段，最多三句：
- 发生了什么
- 这对仓位意味着什么
- 当前动作建议

### 3）宏观 / 事件巡检格式
固定一段，最多四句：
- 有什么事件
- 时间窗口
- 为什么和当前仓位有关
- 当前建议

---

## 六、证据规则

所有建议都必须尽量带证据：
1. 真实仓位：仓位 / 均价 / 浮盈亏 / 杠杆
2. 标的价格位置：最新价 / 24h 高低区间 / 是否贴近低位或高位
3. 主流环境：BTC / ETH 状态、关键位、概率盘倾向
4. 宏观事件：若有则写，若无则明确证据不足

禁止：
- 只给情绪判断
- 只说“感觉”
- 不给依据就喊加减仓

## 八、冲突处理优先级
1. 真实仓位风险
2. 用户个人规则
3. 主巡检完整结论
4. 宏观 / 事件预警
5. 高频轻巡检即时波动

原则：先保命，再考虑机会。
